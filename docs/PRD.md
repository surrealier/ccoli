# PRD (Product Requirements Document)

## 1. 제품 개요
본 프로젝트는 Python 기반 서버와 M5Stack Atom Echo(ESP32) 클라이언트 간 실시간 음성 통신으로 동작하는 음성 인터랙션 시스템입니다.

- 클라이언트: 음성 캡처, VAD 기반 스트리밍, 서버 명령 수신
- 서버: STT 처리, 명령 파싱/대화 처리, 클라이언트 제어 명령 반환

## 2. 목표
- 한국어 음성 입력을 안정적으로 인식(STT)하고 의도 기반 명령으로 변환
- 로봇 제어 모드와 에이전트 대화 모드를 지원
- 실제 하드웨어와 Docker 기반 테스트 환경 모두에서 재현 가능한 개발 체계 제공

## 3. 시스템 범위

### In Scope
- TCP 기반 서버-클라이언트 프로토콜
- 오디오 스트리밍(16kHz, PCM16LE)
- 명령(JSON) 기반 액션 전달
- 모드 전환(robot/agent), 정보 서비스, 로깅, 메모리/스케줄러
- Docker 기반 테스트 실행 환경

### Out of Scope (현재)
- 완전한 오프라인 TTS 품질 최적화
- 다중 디바이스 동시 제어 최적화
- 펌웨어 OTA 파이프라인

## 4. 주요 사용자 시나리오
1. 사용자가 Atom Echo에 말한다.
2. ESP32가 VAD로 발화 구간을 잘라 서버로 전송한다.
3. 서버가 STT 후 명령/응답을 생성한다.
4. 서버가 JSON 명령 또는 오디오 응답을 ESP32로 전송한다.

## 5. 기능 요구사항

### FR-1. 음성 수집 및 스트리밍
- ESP32는 발화 시작/종료를 감지해 START/AUDIO/END 패킷 전송
- pre-roll 및 keepalive(PING) 지원

### FR-2. STT 및 텍스트 정제
- 서버는 수신 오디오를 정규화 후 STT 수행
- 인식 결과 정제(clean_text), 품질 검증(qc) 적용

### FR-3. 명령 파싱 및 액션 실행
- 로봇 모드에서 각도/방향 명령을 액션(JSON)으로 변환
- 인식 불확실 시 NOOP 또는 정책 기반 fallback

### FR-4. 에이전트 기능
- 대화 응답, 감정 상태, 정보 조회(시간/날씨/뉴스), 일정 관리

### FR-5. 설정 및 로깅
- YAML/.env 기반 설정
- 성능/에러 로깅 및 파일 저장

### FR-6. Docker 테스트 실행
- 서버 테스트는 Docker에서 실행 가능해야 함
- 프로토콜 계약 테스트(패킷/JSON 스키마)를 자동화

### FR-7. 자율형 코딩 루프
- Docker 테스트 게이트를 통과할 때까지 반복 실행 가능한 루프를 제공
- 루프 설정/로그 산출물을 저장소에 남겨 재현 가능해야 함

## 6. 프로토콜 요구사항

### 패킷 형식
- 헤더: `ptype(1 byte) + payload_len(2 bytes, little-endian)`

### 타입 정의
- ESP32 -> Server
  - `0x01`: START
  - `0x02`: AUDIO
  - `0x03`: END
  - `0x10`: PING
- Server -> ESP32
  - `0x11`: CMD(JSON)
  - `0x12`: AUDIO_OUT
  - `0x1F`: PONG

### CMD JSON 최소 필드(권장)
- `action` (string)
- `sid` (int)
- 선택: `angle`, `meaningful`, `recognized`, 기타 모드별 파라미터

## 7. 비기능 요구사항
- 성능: 일반 대화 기준 수 초 내 응답
- 안정성: 연결 끊김/타임아웃 처리 및 재시도
- 관측성: 로그를 통한 오류 분석 가능
- 보안: 민감정보 분리, 샘플 설정 파일 제공

## 8. 하드웨어/공식 스펙 준수 요구사항
- ESP32/M5Stack 관련 코드 변경 시 가능하면 공식 문서 스펙(핀/버퍼/주파수/I2S 제한)을 기준으로 검토
- Docker 시뮬레이터에서도 동일 제약(버퍼 크기/프레임 단위)을 반영
- 스펙 기반 결정 사항은 문서 또는 PR에 근거를 남김

## 9. 검증 전략
- 단위 테스트: 텍스트 처리, 패킷 인코딩/디코딩, 설정 로딩
- 통합 테스트: 시뮬레이터 클라이언트와 서버 간 end-to-end 시나리오
- 회귀 테스트: 프로토콜 변경 시 Golden test 갱신
